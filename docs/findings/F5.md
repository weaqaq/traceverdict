# F-5 — Strict cost reconciliation caught a reasoning-token partition bug

On 2026-07-14, the first paid MiMo task completed 14 model calls and produced a patch, but strict reconciliation found a cost shortfall of `$0.00009033822788733483`. The difference was exactly 306 reasoning tokens at the official output rate.

The raw provider usage satisfied `total_tokens = prompt_tokens + completion_tokens` and `reasoning_tokens <= completion_tokens`. LiteLLM partitioned completion into disjoint text and reasoning components, while the initial custom registry assigned zero price to the reasoning partition. The fixed registry prices both disjoint partitions at the same official output rate, counting every generated token once—not twice.

One unified rule exactly reconstructs both real datasets used in the audit: the two-call synthetic probe total and the first paid task's official currency total. The stopped run remains in the private ledger as a paid non-sample; a new immutable config version was used for accepted probe runs.

The practical result is the point of strict mode: a functioning model and plausible patch were not allowed to hide a provider accounting mismatch.
